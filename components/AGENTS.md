# Module Context
- React UI 구성 요소 집합: 설정 패널, 출력/히스토리/롱폼/템플릿/유튜브 검색, 마스터 스튜디오, 토스트 등 화면 요소 관리.
- 공용 버튼/아이콘(lucide-react) 및 레이아웃 컴포넌트, 모달/토스트/패널 단위로 구성.

# Tech Stack & Constraints
- TypeScript + React 함수형 컴포넌트, Vite 빌드. CSS/스타일링은 기존 패턴(모듈/전역 스타일) 유지, 인라인 하드코드 색상 최소화.
- Props/State는 명시적 타입 사용(`types.ts` 재사용), 불필요한 any 금지.
- 외부 효과(스토리지/네트워크)는 서비스/훅을 통해 위임, 컴포넌트는 표시·이벤트 위주.

# Implementation Patterns
- 상태는 상위(App) 또는 훅으로 위임, 컴포넌트는 제어형 입력/콜백 기반으로 유지.
- 리스트/패널 렌더링 시 key 안정성 확보, 로딩/에러/빈 상태 UI 명시.
- 이미지/영상/템플릿 결과 표시 시 기존 DTO 구조(스토리, 템플릿 결과)를 그대로 사용.
- 토스트/모달 등 글로벌 UI는 단일 호스트 컴포넌트에 모아 중복 마운트 방지.

# Testing Strategy
- 기본 확인: `npm run dev` 실행 후 주요 플로우 수동 점검(대본 생성, 템플릿 분석, 히스토리 로드, 이미지/영상 저장 표시).
- 성능/메모리: 대량 리스트 렌더링 시 useMemo/useCallback으로 불필요한 리렌더 체크.
- 시각 회귀가 우려될 때 Playwright 시나리오를 추가 실행(가능한 환경에서).

# Local Golden Rules (Do/Don'ts)
- Do: 새 컴포넌트는 접근성 속성(aria-label 등)과 오류/로딩 상태를 명확히 포함.
- Do: 네트워크 호출은 서비스 함수로 위임, UI에서는 호출 결과만 처리.
- Don't: 전역 mutable 변수나 window 직접 접근 남발 금지(필요 시 훅으로 캡슐화).
- Don't: 서버 포맷 가정한 매직 문자열 하드코딩 금지(상수/타입 사용).
